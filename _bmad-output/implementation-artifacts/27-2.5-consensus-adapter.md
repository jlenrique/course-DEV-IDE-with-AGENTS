## Acceptance Criteria

### Behavioral (AC-B.*)

1. **AC-B.1 — `ConsensusProvider(RetrievalAdapter)` subclass ships.** New module `skills/bmad-agent-texas/scripts/retrieval/consensus_provider.py`. Class `ConsensusProvider` subclasses `RetrievalAdapter`; declares `PROVIDER_INFO: ClassVar[ProviderInfo]` with `id="consensus"`, `shape="retrieval"`, `status="ready"`, `auth_env_vars=["CONSENSUS_API_KEY"]`, `spec_ref` pointing at this file, `capabilities` extending the placeholder's list (adds `"evidence-synthesis"` + `"study-design-tagging"` if Consensus surfaces these). On module import, `__init_subclass__` auto-registers via `provider_directory.register_adapter`, superseding the `consensus: ratified` placeholder; `list_providers()` now shows `consensus: ready`.

2. **AC-B.2 — Seven abstract methods implemented per 27-0 ABC contract.**
   - `formulate_query(intent: RetrievalIntent) -> dict` — deterministic translation of `intent.intent` + `provider_hints[consensus].params` into a Consensus query dict (`{mode: "search" | "paper", query: str | doi: str, filters: {...}}`). Byte-identical across invocations with same input (AC-T.2 determinism self-test, mirrors scite pattern).
   - `execute(query: dict) -> list[dict]` — dispatches to `MCPClient.call_tool("consensus", tool_name, args)` with Bearer auth resolved lazily from `CONSENSUS_API_KEY`. Raises `MCPAuthError` / `MCPRateLimitError` / `MCPFetchError` / `MCPProtocolError` unchanged (dispatcher wraps). No retry logic inside adapter.
   - `apply_mechanical(results, criteria) -> filtered` — honored keys: `date_range` ([start, end] YYYY-MM-DD; row.date within), `min_results` (passthrough; dispatcher post-evaluates count), `exclude_ids` (DOI or `consensus_paper_id`), `license_allow` (if Consensus surfaces license flag).
   - `apply_provider_scored(results, criteria) -> filtered` — honored keys: `consensus_score_min` (Consensus-native agreement-strength filter), `study_design_allow` (whitelist of study-design tags, e.g., `["meta-analysis", "rct"]`), `sample_size_min` (if Consensus surfaces `sample_size` at the row level).
   - `normalize(results) -> list[TexasRow]` — Consensus paper → canonical `TexasRow`. `title`, `body` (abstract; full text generally not surfaced by Consensus), `authors`, `date` from metadata. `provider_metadata.consensus` sub-object holds Consensus-specific fields (see AC-B.3).
   - `refine(previous_query, previous_results, criteria) -> dict | None` — monotonically-loosening. Uses `refinement_registry.drop_filters_in_order` with Consensus-specific order: `sample_size_min` → `consensus_score_min` → `study_design_allow` → `date_range`. If PDG-3-extended resolves scope, additional strategies deferred to `27-2.5-refinement-hardening` follow-on.
  - `identity_key(row: TexasRow) -> str` — **DOI-primary** (critical for cross-validation with scite — AC-B.8). `row.provider_metadata["consensus"]["doi"]` (primary); fallback `row.provider_metadata["consensus"]["consensus_paper_id"]` if DOI missing; fallback `row.source_id` as last resort. Raises `NotImplementedError` ONLY if none available.

3. **AC-B.3 — `declare_honored_criteria` returns explicit key set.** Returns `frozenset({"date_range", "min_results", "exclude_ids", "consensus_score_min", "study_design_allow", "sample_size_min", "license_allow"})`. Unknown keys log to `refinement_log` per 27-0 AC-B.2 strengthened contract.

4. **AC-B.4 — `provider_metadata.consensus` sub-object schema.** Fields populated on every `TexasRow` produced by `normalize`:
   - `doi: str | null` (primary identity key)
  - `consensus_paper_id: str | null` (Consensus internal ID; present when surfaced)
   - `title: str`, `authors: list[str]`, `year: int | null`
   - `venue: str | null`, `abstract: str`
   - `consensus_score: float | null` — agreement strength (0.0–1.0; semantic meaning per Consensus docs)
   - `study_design_tag: str | null` — e.g., `"meta-analysis"`, `"rct"`, `"cohort"`, `"review"`, `"other"`
   - `sample_size: int | null` — aggregate sample size if surfaced
   - `evidence_strength: str | null` — e.g., `"strong"`, `"moderate"`, `"weak"` if surfaced
   - `consensus_url: str` — link to Consensus report for the paper
   - `known_losses: list[str]` — sentinels (e.g., `["abstract_only"]` when full body unavailable, `["study_design_unknown"]` when tag not surfaced)

   **Schema doc-layering obligation (AC-C.4 inherited from 27-2 pattern):** Authoritative schema lives in `extraction-report-schema.md` under the existing `## Provider Metadata Sub-objects` H2 (added in 27-2). Add a new H3 `### provider_metadata.consensus` subsection listing all AC-B.4 fields with types, nullability, ranges, and a worked example. `retrieval-contract.md` "For Tracy" gets a pointer-only subsection (~15 lines) describing Consensus signals at a glance + link to the schema. NO duplicate schema table.

5. **AC-B.5 — Controlled failure on Consensus-specific errors.** MCPClient exceptions flow through unchanged: auth failure (401/403) → `MCPAuthError` → dispatcher records `ProviderResult` with `acceptance_met: False` + `refinement_log` entry `reason="auth_failed"`. Rate-limit (429) → `MCPRateLimitError` → `reason="rate_limited"`. Generic fetch/timeout → `MCPFetchError` → `reason="fetch_failed"`. Malformed JSON-RPC → `MCPProtocolError` → `reason="protocol_error"`. No traceback leaks to stdout (handled by MCPClient). Missing `CONSENSUS_API_KEY` → `MCPAuthError` at first `call_tool` invocation (lazy resolution per MCPClient design, mirrors scite pattern).

6. **AC-B.6 — Bearer auth integration.** ConsensusProvider wires `MCPClient` with `MCPServerConfig(url=<consensus_mcp_url>, auth_env=["CONSENSUS_API_KEY"], auth_style="bearer")` (contrasts with scite's `auth_style="basic"`). Verify via unit test `test_consensus_execute_passes_bearer_header` that `Authorization: Bearer <token>` appears in the outbound HTTP request.

7. **AC-B.7 — Partial graceful degradation on missing Consensus-specific fields.** When Consensus response lacks `study_design_tag`, `sample_size`, or `evidence_strength` (all optional per current Consensus API surface knowledge), `normalize` emits the row with those fields as `null` and adds an appropriate sentinel to `known_losses` (e.g., `"study_design_unknown"`). Does NOT raise; does NOT filter the row out — downstream (Tracy, Vera) decides what to do with under-annotated rows.

8. **AC-B.8 — Cross-validation with scite (FIRST REAL EXERCISE — THE NOVELTY OF THIS STORY).** When `RetrievalIntent` has `cross_validate: true` AND `provider_hints` contains both `scite` and `consensus` entries, the dispatcher (27-0's existing multi-provider code path):
   - Fans out to both providers independently under single-provider rules.
   - Merges results by `identity_key(row)` — DOI-primary for both adapters.
  - Annotates each merged row with `ConvergenceSignal` populated per 27-0 contracts.py schema: `providers_agreeing: list[str]` (provider IDs that returned this identity key), `providers_disagreeing: list[str]` (empty in v1 — only structural disagreement modeled), `single_source_only: list[str]` (provider IDs that returned this identity key without corroborating providers).
   - Returns `list[ProviderResult]` (one per provider) with merged rows surfaced; `ConvergenceSignal` is structural only — tells you whether both providers returned the same DOI, NOT whether they semantically agree.

   **Spec-level clarification**: `ConvergenceSignal` = "structural agreement by identity key." If scite says "DOI X is supporting citation" and Consensus says "DOI X has low consensus_score," the signal is `providers_agreeing: ["scite", "consensus"]` — the DOI converged, even though the two providers' semantic framings differ. Semantic reconciliation is Tracy's job downstream.

### Architecture guardrails (inherited from 27-0 + 27-2)

- **Consensus is a `RetrievalAdapter` subclass, NOT a standalone client.** Lives in `retrieval/` alongside `scite_provider.py` and `fake_provider.py`. `MCPClient` supplies HTTP; ConsensusProvider supplies provider-DSL translation.

- **Refinement is monotonically-loosening.** Consensus key-order `["sample_size_min", "consensus_score_min", "study_design_allow", "date_range"]`. Exhaustion returns `None`.

- **`convergence_signal` is STRUCTURAL.** Providers agreeing on DOI ≠ providers agreeing on semantic framing. Tracy (Epic 28) handles semantic reconciliation. Documented in AC-B.8 spec-level clarification + AC-T.12 atom 3 test docstring.
- **Identity-key DOI-primary for cross-validation to work with scite.** Both adapters return DOI as primary identity; merge works. If either falls back to provider-specific ID (preprint case), the row appears in only one provider's result (single_source_only → true). That's the designed behavior, not a bug.
- **Consensus-specific fields live under `provider_metadata.consensus` opaque sub-object.** Do NOT surface `consensus_score` / `study_design_tag` / `sample_size` / `evidence_strength` at the top-level `TexasRow` schema. Per-provider fields do not pollute the canonical schema (same discipline as scite's `provider_metadata.scite`).

## Cross-Validation: Worked Example (for Tracy authors reading this spec)

After 27-2.5 ships, Tracy can emit a `RetrievalIntent` like:

```yaml
run_id: TRIAL-2-C1-M1-EVIDENCE-BOLSTER-001
sources:
  - ref_id: src-evidence-001
    role: primary
    intent: "peer-reviewed studies on intermittent fasting and cardiovascular risk, published 2020-2026"
    provider_hints:
      - {provider: "scite", params: {mode: "search"}}
      - {provider: "consensus", params: {mode: "search"}}
    cross_validate: true
    acceptance_criteria:
      mechanical:
        date_range: ["2020-01-01", "2026-12-31"]
        min_results: 8
      provider_scored:
        authority_tier_min: "peer-reviewed"        # scite-honored
        supporting_citations_min: 20                # scite-honored
        consensus_score_min: 0.6                    # consensus-honored
        study_design_allow: ["meta-analysis", "rct"]  # consensus-honored
      semantic_deferred: "filter for studies with explicit hypothesis testing on CV endpoints"
    iteration_budget: 3
    convergence_required: true
```

Dispatcher fans out:
- SciteProvider honors `authority_tier_min` + `supporting_citations_min` (ignores `consensus_score_min`, `study_design_allow` — they're not in its `declare_honored_criteria`); returns list of DOIs with smart-citation classification populated.
- ConsensusProvider honors `consensus_score_min` + `study_design_allow` (ignores `authority_tier_min`, `supporting_citations_min`); returns list of DOIs with research-synthesis signals populated.

Dispatcher merges by `identity_key` (DOI):
- DOI-X in both → single merged row; `provider_metadata.scite` + `provider_metadata.consensus` both populated; `convergence_signal.providers_agreeing: ["consensus", "scite"]`.
- DOI-Y in scite only → row with only `provider_metadata.scite`; `convergence_signal.single_source_only: true; providers_agreeing: ["scite"]`.
- DOI-Z in Consensus only → row with only `provider_metadata.consensus`; `convergence_signal.single_source_only: true; providers_agreeing: ["consensus"]`.

Tracy's post-fetch pass (Epic 28) reads `convergence_signal` + both `provider_metadata` sub-objects to produce editorial judgments. Evidence-bolster control (Sprint #1 story) toggles the `cross_validate: true` flag based on operator intent.

This worked example becomes part of the `docs/dev-guide/how-to-add-a-retrieval-provider.md` "Cross-validation in practice" subsection after 27-2.5 closes.
